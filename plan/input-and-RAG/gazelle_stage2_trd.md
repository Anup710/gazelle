# Gazelle — Stage 2 TRD
## RAG Pipeline (Query → Retrieve → LLM Response)

# Objective

Implement a production-simple but high-quality RAG pipeline for educational video tutoring.

The pipeline should:
- support multilingual querying
- maintain session isolation
- support conversational continuity
- optimize for grounded retrieval quality
- remain operationally simple

---

# Existing Stack Dependencies

Inherited from earlier stages:

| Component | Technology |
|---|---|
| Backend | FastAPI |
| Metadata DB | Supabase Postgres |
| Vector DB | Qdrant |
| Embeddings | OpenAI text-embedding-3-small |
| STT (video + voice queries) | Groq Whisper |
| Generation LLM | OpenAI GPT-4o or similar |
| Transcript Source | Structured transcript JSON |

Design choice: All inference (embeddings, augmentation, generation) stays within the OpenAI ecosystem for consistency. Groq Whisper is the only external provider, used exclusively for STT.

# Query Augmentation Layer

## Purpose

Improve retrieval quality before embedding generation.

This is intentionally the only retrieval-quality enhancement introduced in V1.

---

# Augmentation Goals

The augmentation layer should:
- expand shorthand educational questions
- resolve conversational ambiguity
- improve semantic completeness
- preserve educational intent
- preserve user language

Input:
```text

```

Conversation Context:
```text

```

Augmented Query:

# Implementation Strategy

Use a lightweight LLM call before embeddings.

Recommended behavior:
- deterministic
- low temperature
- concise augmentation

Recommended parameters:
```python
temperature = 0.2
max_tokens = 120
```

---

# Important Constraint

The augmentation layer should NOT:
- answer the question
- hallucinate missing context
- change user intent

It only reformulates the retrieval query.

# Conversation Context Handling

## Architecture

Chat history is **client-managed**. The server is stateless — it does not store or persist conversation turns. The client sends context with each request.

This eliminates:
- conversations table in Supabase
- server-side session state management
- cleanup/expiry logic

Tradeoff: conversation is lost on page refresh (acceptable for V1).

---

# Two-Tier Context Strategy

Conversation context uses two layers to balance quality and context window size:

### Tier 1 — Conversation Summary
A compressed summary of all turns before the sliding window. Generated by an LLM compaction call every 4 turns. Appended to the system prompt.

### Tier 2 — Recent Turns (Sliding Window)
Last 4 turns in full verbatim detail.

```python
last_n_turns = 4
compaction_interval = 4  # generate summary every 4 turns
```

### Prompt Structure

```text
System Prompt
    + Conversation Summary (compacted history)
    + Retrieved Chunks (top 5)
    + Last 4 turns (full)
    + Current query
```

### Compaction Behavior

- Every 4th turn, the server generates a summary of the conversation so far
- Uses GPT-4o-mini with low temperature, concise output
- Summary is returned to the client as part of the response
- Client stores and sends the summary with subsequent requests
- Summary replaces all turns older than the sliding window

```python
# Compaction call parameters
temperature = 0.2
max_tokens = 150
```

### Why Two-Tier

- Recent turns preserve exact wording (critical for follow-ups like "explain that again")
- Summary preserves topic drift and earlier conclusions without verbatim repetition
- Context window stays bounded regardless of conversation length
- Compaction cost is amortized (one extra LLM call every 4 turns)

# Session Isolation

All retrieval queries MUST be scoped to:
```python
session_id
```

Example filter:
```python
{
  "must": [
    {
      "key": "session_id",
      "match": {
        "value": session_id
      }
    }
  ]
}
```

No cross-session retrieval is allowed.

# Hallucination Reduction

Hallucination prevention is handled through multiple layers.

---

# Layer 1 — Retrieval Grounding

The model is instructed to answer primarily from:
- retrieved context

---

# Layer 2 — Session Isolation

Prevents unrelated content leakage.

---

# Layer 3 — Controlled Context Size

Only:
- top 5 chunks

This reduces:
- context dilution
- contradictory information
- prompt confusion

---

# Layer 4 — Refusal Instruction

The prompt explicitly instructs:
- refusal when evidence is insufficient

---

# Layer 5 — Similarity Threshold (Required)

```python
MIN_SIMILARITY_SCORE = 0.72
```

If no chunk crosses threshold:
- return insufficient information response
- skip generation

This is one of the strongest systematic anti-hallucination controls and is required, not optional. The exact threshold value is a hyperparameter — 0.72 is a starting point, and the final value should be decided through experimentation.

---

# Layer 6 — Prompt Restriction

Recommended instruction:
```text
Do not answer beyond the retrieved context.
If the information is unavailable, explicitly state that.
```

# Generation LLM

## Model
OpenAI GPT-4o or similar

## Reasoning
- stays within OpenAI ecosystem (same provider as embeddings)
- multilingual fluency (English, Hindi, Hinglish)
- low latency, low cost
- strong instruction following
- good educational tone
- low hallucination tendency

GPT-4o-mini is used for all LLM calls in this stage: query augmentation, conversation compaction, and response generation.

# Request

```json
{
  "session_id": "...",
  "query_text": "...",
  "conversation_summary": "Previously discussed Newton's laws and their applications...",
  "recent_turns": [
    { "role": "user", "content": "..." },
    { "role": "assistant", "content": "..." }
  ]
}
```

Notes:
- `conversation_summary` is empty string on first query, updated by client every 4 turns
- `recent_turns` contains the last 4 turns (up to 8 messages)

---

# Response

```json
{
  "response_text": "...",
  "language": "english",
  "conversation_summary": "...",
  "sources": [
    {
      "chunk_id": "...",
      "start_time": 120.5,
      "end_time": 142.9
    }
  ]
}
```

Notes:
- `conversation_summary` is included only on compaction turns (every 4th turn), null otherwise
- Client replaces its stored summary when a new one is returned
